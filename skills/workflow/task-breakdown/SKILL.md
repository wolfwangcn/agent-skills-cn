# 任务拆解 (Task Breakdown)

把复杂任务分解为有序、可验证的小步骤。

## 指令

在写任何代码或执行任何操作之前，先进入只读模式：
1. 理解全貌
2. 找出依赖关系
3. 产出计划

**计划阶段不执行任何操作。** 输出是一份结构化计划。

## 流程

### 第一步：评估

- 这个任务能一步完成吗？→ 能就直接做，不需要本 skill
- 需要多步 → 继续

### 第二步：画依赖图

找出什么依赖什么。先做地基，再盖楼：

```
基础设施/数据
    │
    ├── 核心逻辑
    │       │
    │       ├── 接口/API
    │       │       │
    │       │       └── 展示层/UI
    │       │
    │       └── 验证/测试
    │
    └── 配置/文档
```

### 第三步：垂直切片

**好的拆法（垂直）：** 每步交付一个完整可用的小功能
**坏的拆法（水平）：** 先做完所有数据库，再做完所有接口，再做完所有 UI

### 第四步：写任务清单

每个任务包含：

```markdown
## 任务 [N]：[动词开头的标题]

**做什么：** 一句话说明
**完成标准：**
- [ ] [具体可验证的条件]
- [ ] [具体可验证的条件]
**验证方式：** [怎么确认做对了]
**依赖：** [依赖哪些前置任务]
**预估：** S/M/L
```

### 第五步：排序 + 检查点

- 高风险的排前面（fail fast）
- 每 2-3 个任务设一个检查点
- 检查点确认一切正常后再继续

## 任务大小参考

| 大小 | 文件数 | 示例 |
|------|--------|------|
| **S** | 1-2 | 添加一个接口、改一个配置 |
| **M** | 3-5 | 一个完整功能切片 |
| **L** | 5+ | ⚠️ 太大了，继续拆 |

## 规则

- 步骤 3-7 个（超过 7 个就分阶段）
- 每步用动词开头（创建、配置、测试、部署）
- 每步做完后系统应处于可工作状态
- 遇到问题立即暂停，不跳步
- 输出计划后等用户确认再开始执行
